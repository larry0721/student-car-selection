# Current System Audit

Audit date: July 12, 2026

This document describes the current working version of the Student Car Selection app before the next AI advisor improvement phase. It is a snapshot of how the app works today. It does not propose product or visual changes.

## Current App Summary

The app is a Next.js, TypeScript, and Tailwind CSS web app for first-time car buyers. Users answer optional budget, lifestyle, ownership-cost, safety, drivetrain, body-style, climate, and preference questions. The app filters vehicles that do not satisfy strict requirements, scores the remaining cars, shows the top three recommendations first, and lets users open details or compare cars side by side.

The app currently runs from a local processed vehicle catalog plus optional overlays from online APIs and user-uploaded CSV files. Supabase schema and client helpers exist, but the current frontend does not persist user profiles, saved comparisons, or recommendation sessions to Supabase yet.

## Frontend Structure

The frontend is built with the Next.js App Router.

- `app/layout.tsx` defines the root HTML metadata and imports global styles.
- `app/page.tsx` renders the main advisor experience.
- `app/globals.css` contains global Tailwind styling.
- `components/BuyerProfilePlanner.tsx` is the main client component. It holds the active view, user profile answers, uploaded CSV overlays, online data overlays, AI recommendation summaries, search state, and compare list.
- `components/RecommendationCard.tsx` renders each recommendation card, compact summary, expandable details, score breakdown, data source/date, cost estimates, common issues, alternatives, and buying tips.
- `components/DataImportPanel.tsx` renders the optional CSV upload and online data loading controls.

The main app has three views:

- `Advisor`: questionnaire, written requirements box, search button, and top recommendations.
- `Compare`: side-by-side comparison table for selected cars, or the top three by default.
- `Advanced options`: data source controls, CSV import, provider statuses, and scoring weight controls.

The questionnaire is optional. A user can leave many fields unchanged and still get recommendations from the default profile.

## Backend Structure

The backend is made of Next.js route handlers under `app/api`.

- `app/api/profile-intake/route.ts` parses the user's written requirements into structured profile fields. It uses OpenAI when `OPENAI_API_KEY` is configured and falls back to a local keyword parser when OpenAI is unavailable.
- `app/api/recommendations/route.ts` generates personalized recommendation text for already-scored vehicles. It uses OpenAI when configured and falls back to local summaries when OpenAI is unavailable.
- `app/api/vehicle-data/enrich/route.ts` enriches shortlist vehicles with NHTSA, FuelEconomy.gov, and optional used-car listing API overlays.

The API routes do not currently write records to a database. They receive request data, return JSON, and let the browser keep the current session state.

## Where Car Data Comes From

The base catalog is loaded from:

- `data/processed/vehicleCatalog.json`
- `data/processed/vehicleCatalog.metadata.json`
- `data/vehicleCatalog.ts`

The processed catalog was generated from raw Kaggle datasets reviewed in `data/raw/KAGGLE_DATASET_REVIEW.md`. The metadata says the catalog was generated on July 8, 2026 from these sources:

- `data/raw/craigslist-carstrucks-data/vehicles.csv`
- `data/raw/car-features-msrp/data.csv`
- `data/raw/usedcarscatalog/cars.csv`
- `data/raw/2023-car-model-dataset/Car_Models.csv`
- `data/raw/electric-vehicle-specs-2025/electric_vehicles_spec_2025.csv.csv`

The current catalog has 320 vehicle records. It combines used price/mileage/listing-style data, vehicle specs, body styles, drivetrain coverage, transmission coverage, newer model coverage, and EV coverage.

Online overlays can come from:

- NHTSA vPIC public API through `lib/data/nhtsa.ts`.
- FuelEconomy.gov public API through `lib/data/fuelEconomy.ts`.
- A custom used-car listing API or Marketcheck through `lib/data/usedCarListings.ts`.

User-uploaded overlays can come from CSV files through:

- `components/DataImportPanel.tsx`
- `lib/data/csvImport.ts`

Supported CSV columns are `make`, `model`, `year`, `reliability`, `safety`, `insurance`, `maintenance`, `mpg`, `commonIssues`, `imageUrl`, and `listingUrl`.

## How Data Is Cleaned

The processed catalog is generated by `scripts/build_vehicle_catalog.py`. That script normalizes raw vehicle rows into a shared vehicle shape. It maps model aliases, body types, drivetrain, transmission, fuel type, reliability heuristics, resale heuristics, safety heuristics, insurance estimates, maintenance estimates, depreciation estimates, pros, watchouts, and common issues.

At recommendation time, `lib/data/vehicleValidation.ts` audits every vehicle before it can be recommended. Invalid or suspicious records are excluded from recommendations.

Current validation checks include:

- year must be in a realistic consumer-car range
- make and model must be present and not suspiciously short or ambiguous
- body style must be supported
- known model/body combinations must make sense
- drivetrain, transmission, and fuel type must be supported
- price must be within a realistic recommendation range
- very new vehicles cannot have suspiciously low prices
- mileage must be realistic
- very new vehicles cannot have suspiciously high mileage
- MPG or MPGe must fit the fuel type
- safety and reliability scores must be in realistic ranges
- insurance and maintenance estimates must be in realistic monthly ranges
- seats must fit the vehicle body style
- suspicious drivetrain/model combinations are rejected

`scripts/test_vehicle_validation.mjs` runs the validation audit against the processed catalog and also tests synthetic bad insurance and maintenance estimates.

## How Online and CSV Data Are Merged

`lib/data/mergeVehicleData.ts` applies overlays to the base catalog. Overlays are matched by exact vehicle key and by vehicle family key from `lib/data/vehicleIdentity.ts`.

When an overlay is applied, it can replace or extend fields such as reliability, safety, insurance, maintenance, depreciation, MPG, fuel type, body type, drivetrain, transmission, seats, price, mileage, common issues, pros, watchouts, image URL, listing URL, source list, and updated date.

Every enriched vehicle includes:

- `dataSources`: where the current data came from, such as `seed`, `nhtsa`, `fueleconomy.gov`, `listing-api`, or `csv-import`
- `dataUpdatedAt`: the latest known source date

Recommendation cards display the data source and updated date.

## How Recommendation Scores Are Calculated

The scoring logic lives in `lib/recommendations.ts`.

The default scoring weights are:

- budget fit: 30%
- reliability: 20%
- safety: 15%
- fuel economy: 10%
- insurance/ownership cost: 10%
- performance: 5%
- practicality: 5%
- resale value: 5%

Users can change these weights in Advanced options. The app normalizes the weights so they always add up to 100%.

The app first filters out invalid data records and strict requirement misses. Strict misses include things like price over buying power, insurance over the user's cap, mileage over the user's max, MPG below the minimum, wrong body style, wrong drivetrain, wrong transmission, too few seats, snow climate without AWD/4WD, or safety/reliability below a selected high priority.

After filtering, each vehicle receives category scores:

- affordability
- reliability
- safety
- fuel economy
- ownership cost
- performance
- practicality
- resale value

Practicality is a blended score made from body style, cargo, family fit, drivetrain, climate, mileage, year, transmission, features, modification fit, and purchase-condition fit.

The final overall score is the weighted sum of the category scores. Ties are sorted by lower vehicle price.

Each scored vehicle also receives:

- `matchSummary`: overall, affordability, reliability, safety, ownership cost, practicality
- `scoreBreakdown`: raw category scores
- `weightedContributions`: how much each category contributed to the final score
- `reasons`: specific explanation bullets
- `misses`: warning bullets
- `ownership`: monthly insurance, maintenance, fuel, and annual depreciation
- `firstYearOwnership`: first-year insurance, maintenance, fuel, depreciation, and total
- `similarAlternatives`
- `buyingTips`

## How Buying Power Is Calculated

Buying power is calculated in `lib/affordability.ts`.

The app starts with the user's monthly budget. It subtracts:

- monthly insurance budget
- estimated monthly fuel cost
- maintenance reserve

The remaining amount is called `paymentBudget`, or "payment room" in the UI.

Fuel cost is calculated as:

```text
expected annual mileage / 12 / minimum MPG target * fuel price
```

Maintenance reserve is:

```text
max($75, 10% of monthly budget)
```

The app converts payment room into loan principal using APR and loan term. Then it adds the down payment and divides by 1.08 to roughly account for taxes and fees.

In plain English:

```text
buying power = estimated purchase price supported by payment room, loan term, APR, down payment, and estimated taxes/fees
```

For cash buyers, `lib/recommendations.ts` gives priority to the stated purchase budget when deciding the effective max price.

## How Ownership Costs Are Estimated

Insurance:

- Stored as `vehicle.insurance` in the catalog or CSV/listing overlay.
- Validated in `lib/data/vehicleValidation.ts` as a realistic monthly first-car range.
- Used in affordability and displayed as a monthly estimate.

Fuel:

- Estimated in `lib/recommendations.ts`.
- Formula:

```text
expected annual mileage / 12 / vehicle MPG * fuel price
```

Maintenance:

- Uses `vehicle.maintenanceEstimate` when present.
- Otherwise estimated in `lib/recommendations.ts` from age, mileage, condition, body type, and brand factor.
- Toyota and Honda receive a lower brand factor; some other brands receive a higher factor.
- The minimum estimated monthly maintenance is $75.

Depreciation:

- Uses `vehicle.depreciationEstimate` when present.
- Otherwise estimated in `lib/recommendations.ts` from price, vehicle age, and resale score.
- Newer vehicles use a higher annual depreciation rate than older vehicles.

First-year total ownership cost:

```text
12 months insurance + 12 months maintenance + 12 months fuel + annual depreciation
```

The recommendation cards clearly label these as estimates, not verified listing facts.

## Where Recommendation Explanations Are Generated

There are two explanation layers.

The deterministic explanation layer lives in `lib/recommendations.ts` inside `addFitNotes()`. It creates specific reasons such as:

- price compared with buying-power cap
- insurance compared with insurance cap
- MPG compared with MPG target
- estimated payment
- annual mileage/fuel estimate
- reliability score compared with the user's reliability priority
- safety score compared with the user's safety priority
- body style, drivetrain, climate, cargo, resale, features, and ownership-cost notes

The optional AI explanation layer lives in `app/api/recommendations/route.ts`. It sends the profile and the already-scored vehicles to the OpenAI Responses API. OpenAI only writes summaries, reasons, and watchouts. It does not choose the cars or override the deterministic scores. If OpenAI is not configured or fails, the app uses local fallback recommendation text.

Written user requirements are interpreted in `app/api/profile-intake/route.ts`. That route uses OpenAI when available and falls back to local keyword parsing when unavailable.

## What Data Is Saved About Users

In the current working app, user profile answers, written requirements, imported CSV overlays, online overlays, AI recommendations, and comparison selections are stored in React state in the browser session.

The current app does not persist user data to Supabase from the frontend. Refreshing the page resets the in-memory session.

The repository does include a Supabase schema for future persistence:

- `profiles`
- `buyer_profiles`
- `comparison_items`
- `recommendation_sessions`
- `recommendation_results`
- `score_imports`
- `vehicle_score_overlays`
- normalized vehicle, model, make, source, and listing tables

Row-level security policies are included for user-owned tables.

## Current Known Limitations

- User data is not persisted yet.
- Supabase schema exists but is not wired into the current user flow.
- Safety and reliability scores in the base catalog are conservative heuristics unless CSV overlays or future dedicated sources improve them.
- Live used-car listing photos and prices require provider credentials.
- OpenAI improves natural-language interpretation and summaries, but deterministic code still controls filtering and scoring.
